Chapter 1: Foundation

Each new project setup contains a few steps regardless of the technology used, or any other specifics. Some of them might look strange, btu they follow the best proven practices that would save you numerous hours of troubleshooting and bug hunting.

01 — Repository

1.1. Setup repository

There are a few things you need to do before coding - configure your Git.

1.1.A Configure name & email important

It's very often that you have multiple accounts for your git provider. It's crucial that you configure your local repository to use the appropriate name and email. 

git config user.name # shows your current name
              git config user.email # show your current email

              git config user.name "John Doe" # set name in current repository
              git config user.email "john@doe.com" # set email in current repository

1.1.B Pick git workflow important

Getting this right is important even if you work alone, because of consistency. Any team members should be synchronized upfront on the git workflow to be used. Simple GitFlow would suffice. The decision should be well documented somewhere easily accessible.

1.1.C Protect master branch (enforce PRs) great

Master (main) branch should be protected so no code ends up there directly. This would enforce usage of PRs, which are a necessity if working in a team.

Note: Can be skipped if working alone.

1.1.D Configure pipeline extra

Each git hosting provides pipelines, and we need a very very basic one that just builds the project (typically done in an hour). This would help a lot when doing PRs so that we know the code we’re trying to merge is at least "buildable". Having a pipeline would allow us to extend it in the future with linting, tests and deployment procedures.

1.2. Add .gitignore file important

This one requires no explanation, but is often missed. It should include all app-specific folders (www for Angular, build for React, dist for Vue, etc), along with node_modules and any sensitive containing ones (.aws).
